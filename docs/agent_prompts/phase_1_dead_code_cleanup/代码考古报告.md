
# 代码考古报告: src

## 执行摘要

- **总文件数**: 63
- **总代码行数**: ~10,243 LOC
- **死代码占比**: ~65%
- **已迁移到 FastAPI 的功能占比**: ~60%（路由层面），~30%（数据模型/业务逻辑层面）

## 模块详细分析

### 1. `api/` (Express 路由)
**功能定位**: Express 路由注册层，负责挂载各业务模块的 HTTP 端点。  
**文件列表**: `routes/admin.ts`, auth.ts, essay.ts, learning.ts, query.ts, system.ts, voice.ts；`handlers/stream.ts`, simple-stream.ts。  
**FastAPI 映射**:  
- 已覆盖：`auth` → `routers/auth.py`（但功能降级）；`essay` → `routers/essays.py`；`learning` → `routers/learning.py`；`system` → `routers/system.py`；`voice` → `routers/voice.py` + main.py WebSocket。  
- 缺失：admin.ts 中的管理员后台路由（备份、日志、用户管理、服务监控）在 FastAPI 中完全缺失；query.ts 的词汇查询已被 compat_legacy.py 和 vocab.py 间接覆盖。  
**死代码识别**: 所有 Express 路由壳代码均为框架绑定代码，迁移后可直接废弃。  
**迁移建议**: 废弃  
**技术债务**: 低 — 纯路由胶水层，无独立业务逻辑。

---

### 2. `auth/`
**功能定位**: JWT 认证核心实现，包含 Token 生成/验证/刷新和密码强度校验。  
**文件列表**: jwt.ts, password.ts。  
**FastAPI 映射**: FastAPI 的 auth.py 目前仅实现硬编码 `admin/admin` 登录，返回 `dev-token-{timestamp}`，**无 JWT、无注册、无 Refresh Token、无密码哈希**。  
**死代码识别**: 不是死代码，而是被“降级替代”了。  
**迁移建议**: **保留并迁移** — 旧后端的 JWT 实现（`generateTokenPair`、`verifyRefreshToken`、密码哈希逻辑）远比新后端完整，应作为 P0 迁移项。  
**技术债务**: 高 — 当前 FastAPI 认证是临时演示级实现，存在严重安全与功能缺口。

---

### 3. `controllers/`
**功能定位**: Express 控制器层，封装 HTTP 请求处理与业务逻辑编排。  
**文件列表**: admin.controller.ts, auth.controller.ts, essay.controller.ts, learning.controller.ts, query.controller.ts, system.controller.ts, voice.controller.ts。  
**FastAPI 映射**: FastAPI 将业务逻辑直接写在 router 中，没有独立 controller 层。旧控制器的功能被分散到 `routers/*.py` 和 llm.py / ocr.py / tts.py 中。  
**死代码识别**: 控制器壳（req/res/next 处理）是死代码；但内部业务逻辑（如 learning.controller.ts 的学生档案更新、admin.controller.ts 的备份管理）在 FastAPI 中尚未实现。  
**迁移建议**: 废弃（提取有效业务逻辑到 FastAPI router/service 层）  
**技术债务**: 中 — 需要人工梳理哪些业务逻辑尚未被 FastAPI 覆盖。

---

### 4. `database/`
**功能定位**: SQLite 连接管理与自动迁移（db.ts）。  
**文件列表**: db.ts。  
**FastAPI 映射**: FastAPI 使用 db.py（SQLModel + SQLAlchemy engine），`init_db()` 自动建表。  
**死代码识别**: db.ts 中的 sqlite3 原生封装层是死代码。  
**迁移建议**: **部分保留作为 Schema 参考** — db.ts 中定义了 9 张表的完整 Schema（`users`, `sessions`, `learning_records`, `language_profiles`, `llm_finetune_data`, `students`, `vocabulary`, `essays`, `learning_paths`），而 FastAPI 目前仅实现了 5 张表（`public_vocab_entries`, `user_vocab_queries`, `conversation_events`, `essay_submissions`, `essay_results`），且**缺失 `users`、`sessions`、`students`、`learning_paths`** 等核心表。旧 Schema 应作为数据模型补全的权威参考。  
**技术债务**: **高** — 新旧数据库结构差异巨大，无法直接兼容旧数据；若已存在生产数据，迁移风险极高。

---

### 5. `infra/`
**功能定位**: 基础设施封装，目前仅包含 mDNS 服务发现。  
**文件列表**: mdns.ts。  
**FastAPI 映射**: 无对应实现。  
**死代码识别**: mdns.ts 用于局域网内广播 Softbus 服务端口。当前 FastAPI 架构下，前端通过固定 `localhost:8012` 连接，mDNS 无实际调用方（除 index.ts 启动时注册外）。  
**迁移建议**: 废弃  
**技术债务**: 低 — 过度设计，当前架构不需要。

---

### 6. `managers/`
**功能定位**: 自研管理器框架，负责服务、会话、上下文、配置、备份、清理、Prompt 的生命周期管理。  
**文件列表**: backup-manager.ts, cleanup-manager.ts, config-manager.ts, context-manager.ts, prompt-manager.ts, service-manager.ts, session-manager.ts。  
**FastAPI 映射**:  
- service-manager.ts → 无（FastAPI 直接 Python 调用 LLM/ASR/TTS/OCR）。  
- session-manager.ts → 无（FastAPI 使用匿名 `session_id` 字符串，无持久化会话表）。  
- context-manager.ts → context_store.py（功能替代）。  
- prompt-manager.ts → prompts.py + Jinja2（功能替代，但模板目录不同）。  
- config-manager.ts → settings.py + runtime_config.py（功能替代）。  
**死代码识别**: 除 prompt-manager.ts 的模板路径可作为迁移参考外，其余管理器均为旧架构的自研框架代码，属于死代码。  
**迁移建议**: 废弃（归档 Prompt 模板清单）  
**技术债务**: 中 — 自研框架增加了理解成本，但无外部依赖，删除即可。

**迁移建议**: 废弃（提取 JWT 逻辑到 FastAPI）

### 8. `models/`

## 风险与注意事项

1. **数据库 Schema 不兼容风险**：旧后端有 9 张表，新后端仅有 5 张表，且表结构差异大（如旧 `essays` 表与新 `essay_submissions` + `essay_results` 的分表设计不同）。若已有生产数据，直接切换会导致数据丢失，需要制定专门的 Schema 迁移脚本。
2. **认证降级风险**：FastAPI 当前使用硬编码 `admin/admin` + 简单时间戳 Token，无签名验证。在迁移完成前，任何暴露到公网的部署都存在严重安全隐患。
3. **Prompt 模板目录差异**：旧后端的 prompts 与 FastAPI 的 prompts 是两套模板体系，需确认是否有旧模板尚未迁移，避免前端调用时缺少模板。
4. **测试依赖风险**：test-quick.ts 和 test-integration.ts 仍直接 `import` 旧后端的 `ServiceManager`，删除旧代码前需同步更新或删除这些测试文件。

## 下一步建议

1. **立即补全认证体系（P0）**：将 `auth/jwt.ts` 和 `auth/password.ts` 的逻辑迁移到 FastAPI，替换掉硬编码 `admin/admin`，恢复用户注册、登录、Token 刷新功能。
2. **补全核心数据模型（P0）**：参考 `database/db.ts` 的 Schema 和 `models/*.ts` 的业务逻辑，在 FastAPI 中创建 `users`、`sessions`、`students`、`vocabulary`、`learning_paths` 的 SQLModel 定义，并将 SM-2 算法迁移到 Python。
3. **清理死代码（P1）**：在确认 P0 业务逻辑迁移完成后，删除 src 中标记为死代码的目录（`services/`、`softbus/`、`orchestrator/`、`infra/`、`managers/`、shared、`utils/`、`middleware/`、`api/`、`controllers/`），降低代码熵。
4. **迁移管理员后台（P1）**：将 admin.controller.ts 中的备份、日志查看、用户管理等功能迁移到 FastAPI，补齐运维能力缺口。
5. **统一测试与文档（P2）**：更新 tests 目录中引用旧后端的集成测试，并基于新的代码考古报告更新 `MIGRATION_MILESTONES_FASTAPI.md` 中的迁移状态。
