## 一、规划原则

### 1.1 规划层级

| 层级 | 周期 | 输出 | 参与者 |
|------|------|------|--------|
| **战略规划** | 季度 | 产品路线图 | 管理层 |
| **发布规划** | 月度 | 发布计划 | 产品+技术 |
| **Sprint规划** | 2周 | Sprint Backlog | 全团队 |
| **每日规划** | 1天 | 每日站会 | 开发团队 |

### 1.2 规划输入

- 产品需求 (Product Backlog)
- 技术债务 (Tech Debt Backlog)
- 团队速率 (Velocity)
- 资源可用性
- 外部依赖

## 二、Sprint 规划

### 2.1 规划会议流程

```
Sprint 规划会议 (2-4小时)
    │
    ├── 1. 回顾产品 Backlog (30分钟)
    │   └── 产品负责人讲解优先级最高的故事
    │
    ├── 2. 团队速率回顾 (15分钟)
    │   └── 基于历史数据确定本次 Sprint 容量
    │
    ├── 3. 故事估算 (60分钟)
    │   └── 使用故事点估算选中的故事
    │
    ├── 4. 任务拆解 (60分钟)
    │   └── 将故事拆解为具体任务
    │
    └── 5. 承诺与确认 (15分钟)
        └── 团队确认 Sprint 目标并承诺
```

### 2.2 故事点估算

#### 斐波那契数列

| 点数 | 复杂度 | 参考标准 |
|------|--------|----------|
| 1 | 非常简单 | 修改配置、简单Bug修复 |
| 2 | 简单 | 小功能、简单API |
| 3 | 中等 | 标准功能、需要测试 |
| 5 | 复杂 | 涉及多个模块、需要设计 |
| 8 | 很复杂 | 跨服务、技术挑战 |
| 13 | 极复杂 | 需要调研、高风险 |

#### 估算流程

```
产品负责人讲解故事
        │
        ▼
团队讨论澄清问题
        │
        ▼
每个人独立估算 (扑克牌)
        │
        ▼
同时展示估算
        │
        ▼
┌─────────────────┐
│ 估算差异大?     │
└────────┬────────┘
         │
    ┌────┴────┐
    ▼         ▼
   是         否
    │         │
    ▼         ▼
讨论原因   取平均值
    │         │
    └────┬────┘
         ▼
    记录估算结果
```

## 三、任务调整机制

### 3.1 调整触发条件

| 条件 | 调整类型 | 决策人 |
|------|----------|--------|
| 需求变更 | 范围调整 | 产品负责人 |
| 技术障碍 | 技术方案调整 | 技术负责人 |
| 人员变动 | 容量调整 | Scrum Master |
| 外部依赖 | 计划调整 | 项目经理 |
| 质量风险 | 优先级调整 | 团队共识 |

### 3.2 调整流程

```
发现问题/变更
        │
        ▼
评估影响范围
        │
        ▼
提出调整方案
        │
        ▼
┌─────────────────┐
│ 影响 Sprint 目标? │
└────────┬────────┘
         │
    ┌────┴────┐
    ▼         ▼
   是         否
    │         │
    ▼         ▼
团队决策   直接执行
    │
    ▼
更新 Sprint Backlog
    │
    ▼
通知相关方
```

### 3.3 范围调整原则

#### 可以加入

- 已承诺故事的子任务
- Bug修复 (严重程度 ≥ 中)
- 技术债务 (影响当前 Sprint)

#### 不可以加入

- 新的功能故事
- 低优先级任务
- 未估算的任务

#### 可以移除

- 已完成的任务的子任务
- 阻塞且无法解决的任务
- 优先级降低的任务

## 五、资源管理

### 5.1 容量规划

```
Sprint 容量计算

团队成员: 5人
Sprint天数: 10天
可用天数: 5 × 10 = 50人天

减去:
- 会议时间: -5人天
- 代码审查: -3人天
- 学习培训: -2人天
- 缓冲时间: -5人天

实际容量: 35人天
历史速率: 30故事点

本次 Sprint 承诺: 28-32 故事点
```

### 5.2 资源冲突解决

| 冲突类型 | 解决策略 |
|----------|----------|
| 多任务并行 | 优先级排序，专注高优先级 |
| 人员请假 | 调整范围，或借调资源 |
| 技能不足 | 培训 + 专家支持 |
| 工具限制 | 升级工具，或调整方案 |
