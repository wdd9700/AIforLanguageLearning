## 2. 服务模块资源需求分析

### 2.1 AI推理服务（核心瓶颈）

#### 2.1.1 LLM服务（vLLM）

**技术栈**: vLLM + Qwen2.5-72B / Llama3-70B

| 资源类型 | 开发/测试 | 生产环境 | 大规模部署 |
|---------|----------|---------|-----------|
| **GPU** | RTX 4090 24GB × 1 | A100 80GB × 2 | H100 80GB × 4 |
| **显存** | 24GB | 160GB | 320GB |
| **CPU** | 16核 | 64核 | 128核 |
| **内存** | 64GB | 256GB | 512GB |
| **存储** | 500GB NVMe | 2TB NVMe | 5TB NVMe |

**性能基准**:

| 指标 | 4-bit量化 | 8-bit量化 | FP16 |
|-----|-----------|-----------|------|
| **首Token延迟(TTFT)** | 200ms | 300ms | 500ms |
| **吞吐率(tokens/s)** | 80 | 60 | 40 |
| **并发请求数** | 64 | 48 | 32 |
| **显存占用(70B模型)** | 40GB | 70GB | 140GB |

**计算公式**:
```
所需GPU数 = ceil(模型显存需求 × 并发系数 / 单卡显存)
QPS容量 = (单卡吞吐率 × GPU数量) / 平均输出长度
```

#### 2.1.2 ASR服务（faster-whisper）

**技术栈**: faster-whisper + large-v3

| 资源类型 | 开发/测试 | 生产环境 | 大规模部署 |
|---------|----------|---------|-----------|
| **GPU** | RTX 3060 12GB × 1 | RTX 4090 24GB × 2 | A100 40GB × 2 |
| **显存** | 12GB | 48GB | 80GB |
| **CPU** | 8核 | 32核 | 64核 |
| **内存** | 32GB | 128GB | 256GB |

**性能基准** (faster-whisper large-v3):

| 指标 | CPU模式 | GPU FP16 | GPU INT8 |
|-----|---------|----------|----------|
| **实时率(RTF)** | 0.5x | 8x | 12x |
| **延迟(1分钟音频)** | 120s | 7.5s | 5s |
| **并发路数** | 4 | 32 | 48 |
| **WER(普通话)** | 12% | 10% | 11% |

**容量计算**:
```
单GPU并发路数 = 显存(GB) / 每路显存占用(约0.5GB)
总并发容量 = GPU数量 × 单GPU并发路数
```

#### 2.1.3 TTS服务（Piper/XTTS）

|-----|-------|---------|----------|---------|
| **API Gateway** | 2 | 4核 | 8GB | - |
| **业务服务** | 3 | 4核 | 8GB | - |
| **LLM服务** | 1 | 16核 | 64GB | RTX 4090 24GB × 1 |
| **ASR服务** | 1 | 8核 | 32GB | RTX 3060 12GB × 1 |
| **TTS服务** | 1 | 4核 | 8GB | - |

| **Kafka** | 3 | 8核 | 32GB | - |

**总计**:
- CPU: 94核
- 内存: 348GB
- GPU: RTX 4090 × 1 + RTX 3060 × 1
- 存储: 5TB SSD

### 3.2 场景2：课堂实时交互（高负载）

**场景描述**:
- 50个课堂，每课堂20人 = 1000并发
- 实时ASR+TTS流
- VLM课件分析

**资源需求计算**:

| 服务 | 实例数 | CPU/实例 | 内存/实例 | GPU/实例 |
|-----|-------|---------|----------|---------|
| **API Gateway** | 4 | 8核 | 16GB | - |
| **业务服务** | 10 | 8核 | 16GB | - |
| **WebSocket服务** | 4 | 8核 | 16GB | - |
| **LLM服务** | 2 | 32核 | 128GB | A100 80GB × 2 |
| **ASR服务** | 4 | 16核 | 64GB | RTX 4090 24GB × 2 |
| **TTS服务** | 4 | 8核 | 16GB | RTX 3060 × 2 |
| **VLM服务** | 2 | 16核 | 64GB | A100 40GB × 1 |
| **PostgreSQL** | 4(主+3从) | 16核 | 64GB | - |
| **Redis** | 6(3主3从) | 8核 | 32GB | - |
| **Meilisearch** | 3 | 8核 | 32GB | - |
| **Neo4j** | 2 | 16核 | 64GB | - |
| **Kafka** | 3 | 16核 | 64GB | - |
| **MinIO** | 4 | 16核 | 64GB | - |

**总计**:
- CPU: 512核
- 内存: 2,048GB (2TB)
- GPU: A100 80GB × 4 + RTX 4090 × 8 + RTX 3060 × 8
- 存储: 50TB NVMe
- 网络: 10Gbps+

**关键瓶颈分析**:

1. **ASR并发**: 1000并发用户 × 20%同时发言 = 200路ASR
   - 每GPU支持32路 → 需要7张GPU
   - 建议: RTX 4090 × 8

2. **TTS并发**: 1000并发用户 × 30%同时收听 = 300路TTS
   - Piper CPU模式支持100路/4核 → 需要12组
   - 建议: 4实例 × 8核

3. **LLM QPS**: 1000用户 × 0.1 QPS = 100 QPS
   - 每A100支持约20 QPS → 需要5张A100
   - 建议: A100 80GB × 4 + 队列削峰

### 6.4 自动扩缩容

#### K8s HPA配置

```yaml
# LLM服务HPA
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: llm-service-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: llm-service
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Pods
    pods:
      metric:
        name: vllm_queue_length
      target:
        type: AverageValue
        averageValue: "10"
  - type: Resource
    resource:
      name: nvidia.com/gpu
      target:
        type: Utilization
        averageUtilization: 80
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 60
      policies:
      - type: Pods
        value: 2
        periodSeconds: 60
    scaleDown:
      stabilizationWindowSeconds: 300
      policies:
      - type: Pods
        value: 1
        periodSeconds: 120
```

# 基于课程表的预测扩容
class PredictiveScaler:
    def __init__(self):
        self.schedule = load_class_schedule()
    
    def predict_load(self, timestamp):
        """预测未来30分钟负载"""
        upcoming_classes = self.schedule.get_classes(
            start=timestamp, 
            end=timestamp + timedelta(minutes=30)
        )
        
        predicted_concurrent = sum(
            c.student_count for c in upcoming_classes
        )
        
        # 提前15分钟扩容
        if predicted_concurrent > current_capacity * 0.8:
            self.scale_up(
                replicas=calculate_needed_replicas(predicted_concurrent),
                lead_time=timedelta(minutes=15)
            )

### 7.3 容量规划公式

#### LLM容量
```
所需GPU数 = ceil(目标QPS × 平均输出长度 / (单卡吞吐率 × 利用率))

示例:
目标QPS = 100
平均输出长度 = 500 tokens
单卡吞吐率 = 50 tokens/s
利用率 = 0.8

所需GPU数 = ceil(100 × 500 / (50 × 0.8)) = ceil(50000 / 40) = 1250 → 需多张卡
实际: 使用A100 × 4，吞吐率 = 50 × 4 = 200 tokens/s
可支持QPS = 200 × 0.8 / 500 = 0.32 QPS/路 → 需要批处理优化
```

#### ASR容量
```
所需GPU数 = ceil(并发路数 / 单卡并发能力)

示例:
并发路数 = 200 (1000用户 × 20%发言率)
单卡并发 = 32路 (RTX 4090)

所需GPU数 = ceil(200 / 32) = 7 → 建议8张
```

#### 存储容量
```
年存储需求 = 日活跃用户 × 日均数据量 × 365 × 副本数

示例:
DAU = 10,000
日均数据 = 10MB (语音+日志+文档)
副本数 = 3

年存储 = 10000 × 10MB × 365 × 3 = 109.5TB
建议预留50%增长: 109.5 × 1.5 = 164TB
```
